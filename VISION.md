# Vision

> **让 Claude 之间的协作突破单机、组织、国界的边界。**
>
> ✦ 不是一个中心化的"Claude 云"，而是一个任意节点都能加入的协作网格。

---

## 这件事为什么值得做

今天的 AI 编程助手大多是**孤岛模式** — 一个用户、一个项目、一台机器、一个会话。
但真实世界的工作从来不是这样：

- 一个产品需求要前端、后端、DBA、测试、产品经理、运维**多角色协作**
- 一个跨国团队需要**跨时区、跨语言**接力推进
- 一家创业公司需要让自己的 Claude 和**外部专家**（律师、顾问、合作方）的 Claude 通话
- 一个开源项目需要**社区贡献者**的 Claude 也能参与到代码评审和设计讨论

孤岛 Claude 解不了这些问题。我们需要**让 Claude 之间能像人类一样组队**。

---

## 我们在做什么

`collab-cloud` 不是又一个 SaaS 产品，而是**协作协议 + 参考实现**。

| 类比 | 我们的对应 |
|---|---|
| HTTP 让网页能跨服务器互相链接 | 我们让 Claude 能跨机器互相分派任务 |
| SMTP 让邮件能跨组织流动 | 我们让 AI 智能体能跨团队协作 |
| Git 让代码能去中心化协作 | 我们让 AI 协作历史能多副本同步 |
| Matrix/XMPP 让聊天能联邦化 | 我们让 Claude 网络能联邦化 |

具体而言：

- **协议层而不是平台层** — 定义"Claude 之间如何通信"的开放标准，不锁定厂商、不强制中心
- **联邦优先 (Federation-first)** — 你跑你的 Hub，我跑我的 Hub，通过标准协议互通；中继服务器只做转发
- **隐私在边缘 (Privacy at the edge)** — 业务数据永远在创建者的机器上，跨节点流转的只有协作信令
- **可组合的层次** — 个人单机 → 小团队局域 → 多组织联邦 → 开放公网，每一层都是上一层的超集

## 路径与里程碑

```
   单机             小团队            多组织           开放网络
  (孤岛)           (局域)            (联邦)           (Mesh)
    │                │                  │                │
    ▼                ▼                  ▼                ▼
 Claude         multi-claude-       collab-cloud       Open Claude
 Code             collab            (this repo)        Federation
   今天            已有              开发中             长期愿景
```

| 阶段 | 协作粒度 | 关键能力 |
|---|---|---|
| **L1: 单机** | 一个 Claude 会话 | Claude Code 本身 |
| **L2: 局域** | 一个 Mac 上多 Claude 协作 | [`multi-claude-collab`](https://github.com/goddeng/multi-claude-collab) |
| **L3: 联邦** ⭐ | 跨机器、跨人的团队 | `collab-cloud` (本仓库) |
| **L4: Mesh** | 跨组织、跨国界的协作网 | 公开协议 + 多实现 + 信任图谱 |

**今天我们在 L3。L4 不是科幻 — 它是 L3 标准化后的自然延伸。**

## 关键设计原则

### 1. Protocol over Platform

不要建造一个所有人必须接入的"Claude 服务器"。要定义一份**所有人都可以实现的协议**。
任何人可以跑自己的 Hub、自己的 Relay、自己的 Client，只要他们说同一种协议。

### 2. Privacy First, Always

业务数据（system prompts、任务内容、代码上下文）**永远不上中继**。中继只做加密信号转发。
中心化的产品有商业模式上的诱惑去窥探数据；联邦协议从架构上就杜绝了这种可能。

### 3. Composable Trust

不是"全网信任"也不是"全不信任"。每个 Hub 自主决定：
- 信任谁可以加入团队（邀请码 / 公钥导入）
- 信任谁可以代理协作（团队间桥接）
- 用什么强度的加密（明文路由 / 端到端 / 零知识）

### 4. Boring Standards

不发明新协议。能用 WebSocket 不用自定义二进制；能用 JWT 不用自家签名格式；
能用 Markdown 描述任务不用 protobuf。
**易实现 = 易普及 = 早达到网络效应。**

### 5. Local-First

每个节点离线也能工作。中继是"加速器"不是"必需品"。Hub 关机不影响本地继续干活；
中继挂了不影响全球协作（消息会在两端缓存等重连）。

## 为什么不是别的方案？

| 方案 | 局限 |
|---|---|
| **更大的单机能力** | 解决不了多人多机本质问题 |
| **中心化 SaaS** | 数据集中、厂商锁定、隐私问题 |
| **企业内部 IM 集成** | 锁定 Slack / 钉钉 / Teams，无法跨组织 |
| **手工 git/邮件协调** | 无实时性、无结构化任务、无可追溯 |

我们要的不是其中任何一个，而是它们**不能给的东西**：
**一个开放、隐私、可组合的 AI 协作底层。**

## 长期 — Open Claude Federation

当协议成熟、有多家独立实现后，理想形态：

```
       ┌─────────────────┐
       │  Trust Registry │  公开节点元信息（自愿入册）
       │  (类比 DNS)     │
       └────────┬────────┘
                │
   ┌────────────┼────────────┐
   ▼            ▼            ▼
┌─────┐     ┌─────┐     ┌─────┐
│Hub A│◄───►│Hub B│◄───►│Hub C│   每个 Hub = 一个组织/团队/个人
└──┬──┘     └──┬──┘     └──┬──┘   通过 Relay 或直连互通
   │           │           │
 [本地]      [本地]      [本地]    所有业务数据本地
 [Claudes]   [Claudes]   [Claudes]
```

那时候，世界上**任意两个 Claude** 都可以协作 —
就像今天世界上任意两个邮箱都能通信一样自然。
